# 流程图运行时语义

## 作用

本文件定义流程图运行时对象、生命周期、推理接口、暂停/结束语义以及异常传播规则。它为后续 Core 和 Generator 提供统一目标。

## 基本对象

### FlowChart 定义对象

1. FlowChart 定义对象描述一张流程图的静态结构。
2. 它持有节点、端口和连接关系，但不持有会变化的运行时上下文。
3. 同一张 FlowChart 定义可以同时创建任意多个 Flow 和任意多个 Context。

### Flow 运行时类型

1. 每张流程图可生成一个 `Flow<TContext>` 类型，作为该图的一次运行实例。
2. Flow 持有流程图定义引用，也持有本次运行的通用执行态。
3. `CurrentNodeId`、`IsPaused`、`IsCompleted`、`NodeStates` 这类字段应保留在 Flow 内，因为它们只属于单次 Flow 实例，且各张图之间逻辑一致。
4. 同一张 FlowChart 定义可以同时创建任意多个相互独立的 Flow 实例。

### FlowContext 运行时上下文对象

1. 每张流程图仍可生成一个与该图实例紧密关联的 Context 类型，例如 `QuestIntroContext<TExternalContext>`。
2. Context 持有外部业务上下文 `ExternalContext`，以及该图需要暴露给节点逻辑的图专属上下文数据。
3. Context 不是 Flow 通用生命周期状态的承载体；它不负责替代 `CurrentNodeId`、`IsPaused`、`IsCompleted`、`NodeStates`。
4. Context 的价值主要在于承载业务侧和图专属的数据视图，而不是承载所有 Flow 运行态。

## 入口与重入

1. `event` 节点是合法的 Flow 创建入口。
2. 调用方可根据某个入口创建新的 Flow，并为该 Flow 传入对应的 Context。
3. 入口节点选择、当前节点位置和暂停/结束态都属于该次 Flow 实例自身。
4. 因为 FlowChart 定义不持有运行态，所以同一张图可以任意重入，并可同时存在多个 Flow 实例。

## Flow 持有状态

首版 Flow 至少需要持有以下状态：

1. `context`
   当前 Flow 持有的外部/图专属上下文对象。
2. `entryNodeId`
   本次运行选定的入口节点 id。若未显式指定，可由生成器填入默认入口。
3. `currentNodeId`
   当前所在流程节点 id。若尚未开始或已结束，可为空。
4. `isPaused`
   当前是否处于暂停状态。
5. `isCompleted`
   当前是否已经结束。
6. `nodeStates`
   与节点实例绑定的运行时状态集合，用于保存循环索引、等待起始时间、延迟计时、上次输出等节点局部状态。
7. `stepComputeCache`
   单步推理使用的临时缓存。若实现为字段，其生命周期也应只绑定当前 Flow。

## Context 持有状态

首版 Context 至少需要持有以下状态：

1. `externalContext`
   外部业务侧上下文，由调用方传入。
2. 图专属业务上下文字段
   例如某张图运行时需要共享的任务变量、黑板数据、环境快照或服务访问入口。
3. 节点逻辑需要显式暴露的图专属派生数据
   例如通过生成器收敛出的强类型访问入口。

建议补充但非首版强制的状态：

1. `pauseReason`
   若业务确实需要把暂停原因暴露给外部系统，可放在 Flow 或 Context 的调试层上。
2. `lastTransition`
   用于调试和日志记录。
3. `stepCount`
   更适合作为 Flow 级状态。
4. `debugTrace`
   可按需要挂在 Flow 或专门的调试对象上。

## 节点实例状态

1. 只要某个节点跨步保留状态，该状态就必须进入当前 Flow 的 `nodeStates`，而不能保存在 FlowChart 定义对象上。
2. `Pause` 这类节点可以只改写 `isPaused` 和 `currentNodeId`；若未来扩展出“暂停到指定条件/时间”的节点，则应把等待信息写入对应节点状态。
3. `PauseSeconds` 一类节点应在首次进入时把起始时间或唤醒时间写入节点状态；后续调度时继续读取该状态，而不是重新初始化。
4. `ForEach` 一类节点应把迭代索引、当前元素快照以及必要的枚举载荷保存在当前 Flow 的节点状态中。
5. 节点状态的生命周期应与 Flow 绑定；节点执行结束、循环走完或显式重置时，再清空对应状态。
6. `WaitUntil` 这类节点通常不需要额外节点状态；条件未满足时可直接保持 `currentNodeId` 在当前节点并进入暂停态，等待外部恢复后重新求值。

## Flow 自身约束

1. Flow 可以保留定义引用、节点运行时单例、静态连接表以及当前运行实例的通用生命周期状态。
2. Flow 上允许存在 `currentNodeId`、`isPaused`、`isCompleted`、`nodeStates` 等字段，因为它们只绑定单次运行实例，不会污染其它 Flow。
3. Flow 不应把任何可变状态存回 FlowChart 定义对象。
4. 如果未来需要图专属但非通用的业务数据，再由 Context 承载。

## 流程回路

1. 流程连接允许形成回路。
2. `if`、`while`、`for each`、`break`、`continue` 等控制节点依赖这一能力实现。
3. 允许流程回路不代表允许计算回路；计算连接仍然必须保持无环。

## 推理接口

首版 Flow 运行时应提供以下接口语义：

1. `StepOnce()`
   执行一个最小流程推进单位。
2. `Step(maxSteps)`
   最多连续执行 N 个最小流程推进单位。
3. `RunToCompletion()`
   持续执行，直到 `isCompleted == true`。
4. `RunUntilPaused()`
   持续执行，直到 `isPaused == true` 或 `isCompleted == true`。
5. `RunUntil(predicate)`
   持续执行，直到外部基于当前 Flow 的条件返回成立，或流程暂停，或流程结束。
6. `Resume()`
   清除暂停态并允许后续继续推进。

## 步进语义

1. “一步”指一次可观察的流程推进，而不是单纯一次计算端口求值。
2. 一个流程节点在被调度执行后，可能：
   - 跳转到下一个流程节点。
   - 改写当前 Flow 的暂停状态。
   - 改写当前 Flow 的结束状态。
   - 改写当前 Flow 的节点实例状态。
   - 抛出异常。
3. 计算端口求值属于该步内部行为，不单独暴露为 Flow 的公共步进单位。
4. 一次步进中发生的通用运行态变化，应最终落在当前 Flow 上；业务和图专属数据变化再落在 Context 上。

## 暂停语义

Flow 的暂停来源分为两类：

1. 外部请求暂停
   调用方在运行接口的边界上要求当前 Flow 在合适时机停下。
2. 节点主动暂停
   某个流程节点在执行后要求当前 Flow 停止推进，直到后续再次被唤醒。

当前已落地的等待型控制节点可按以下语义理解：

1. `WaitUntil`
   每次执行时求值条件输入；若条件未满足，则保持停留在当前节点并进入暂停态；条件满足后从 `Then` 出口继续。
2. `PauseSeconds`
   首次进入时记录唤醒时间；在唤醒时间到达前保持停留在当前节点并进入暂停态；时间到达后清理节点状态并从 `Then` 出口继续。

补充约定：

1. 当 `isPaused == true` 时，Flow 不应继续自动推进。
2. 被暂停的 Flow 可以在后续再次调用运行接口后继续推进。
3. `Resume()` 只负责解除暂停；它不应隐式清理节点实例状态。
4. 暂停不等于结束；`isPaused` 与 `isCompleted` 是两个独立状态。

## 结束语义

1. 当 Flow 到达显式结束节点或运行时定义的结束条件时，`isCompleted` 置为 `true`。
2. 已结束的 Flow 不应再继续推进。
3. 已结束的 Flow 可以保留最终 Context 与节点状态，以供外部读取结果或调试。

## 异常传播

1. 推理过程中产生的异常不需要在 Flow 或 Context 内部持久保存。
2. 异常应直接抛给调用 Flow 接口的上层。
3. 首版不要求记录异常历史。

## 中间值缓存

1. 单步内部的计算缓存可作为一次 `StepOnce` 的局部临时对象存在。
2. 若计算缓存以字段形式存在，它也必须只绑定当前 Flow，而不能放在共享的 FlowChart 定义对象上。
3. 缓存不得污染其它 Flow 实例运行态。

## 代码生成建议

1. 每张流程图生成一个静态定义对象、一个 `Flow<TContext>` 类型，以及按需生成的 `Context<TExternalContext>` 类型。
2. 定义对象负责创建 Flow；若该图需要图专属上下文视图，也可同时负责创建 Context。
3. Flow 类型负责持有通用运行态，并通过 `Context` 暴露业务和图专属数据。
4. 已知的有状态节点可继续通过 `Dictionary<uint, FlowChartNodeState>` 挂在当前 Flow 上；后续若要强类型化，也应先论证它是否真的需要进入图专属 Context。
5. 节点 partial class 是按节点定义共享生成的，而不是按流程图实例生成的；如果未来允许自定义有状态节点访问运行时上下文，应补一层共享运行时接口，而不是让节点类直接依赖某一张图的专属 Context 类型。
6. 节点抛出的异常直接透传，不在生成骨架中吞掉。
7. 当前生成的等待型时间语义通过共享运行时接口 `IFlowChartTimeContext.UtcNow` 从 `Context` 读取时间；不实现该接口的 `Context` 无法执行 `PauseSeconds` 这类时间节点。

## 当前状态结论

流程图运行时的核心语义已经进一步收敛：FlowChart 定义保持静态，Flow 作为单次运行实例持有通用执行态，Context 负责承载业务和图专属数据视图。现有机制需要继续澄清的是 Flow 与 Context 的职责边界，而不是流程图文件格式、连线规则或类型系统本身。
