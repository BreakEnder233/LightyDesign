# FlowCharts 子系统

## 职责

FlowCharts 子系统定义流程图领域模型、节点定义协议、流程图实例协议、连接规则和代码生成目标。它是 Core、DesktopHost、FlowChartEditor 和 Generator 的共同业务边界。

## 节点类型约定

1. `event` 节点视为开始节点，通常用于事件监听，不允许有流程输入端口，且至少有一个流程输出端口。
2. `flow` 节点视为中间流程节点，必须恰好有一个流程输入端口，且至少有一个流程输出端口，也可以额外拥有计算端口。
3. `compute` 节点只允许拥有计算端口，可以没有计算输入端口，但必须至少有一个计算输出端口。

## 连线约定

1. 一个流程端口出口只能连接到一个流程端口入口。
2. 一个流程端口入口可以接收多个流程端口出口。
3. 一个计算端口出口可以连接到多个计算端口入口。
4. 一个计算端口入口只能连接到一个计算端口出口。
5. 非法方向、非法基数或计算回路都应被视为保存阻断错误。
6. 连接必须通过节点 id 和端口 id 建立，不通过名称建立。
7. 流程连接允许形成回路，流程控制节点依赖这一能力。
8. 对带类型参数的节点，连线合法性不应简单退化为 `object` 通配，而应通过局部类型推断或显式 `typeArguments` 判断是否存在合法具体化结果。
9. 无法唯一收敛的节点实例可保留诊断状态，但不应直接进入代码导出。

## 代码生成目标

1. 每个节点定义生成一个 partial class，供游戏侧实现执行逻辑或计算逻辑。
2. 每张流程图生成 `Definition` 类型、`Flow<TContext>` 运行时类型以及运行时辅助文件。
3. 流程图定义对象负责描述静态结构和创建运行时对象，并通过 `LDD.cs` 暴露统一访问入口。
4. `Flow<TContext>` 运行时对象负责持有单次运行的生命周期状态、节点实例状态与外部上下文引用，并执行调度。
5. `Flow<TContext>` 运行时对象暴露推理 1 步、推理 N 步、推理直到结束、推理直到暂停、推理直到条件满足等接口。
6. 对需要时间语义的等待型节点，运行时通过共享时间接口从 `Context` 读取当前时间，而不是把时间源写死在生成代码里。

## 当前状态结论

FlowCharts 子系统目前已经完成首版代码生成与 LDD 集成闭环：流程图定义、运行时骨架、宿主导出和统一入口都已打通。后续开发重点应转向 Core 正式模型、泛型节点导出前收敛、自定义有状态节点契约，以及等待型标准节点模板，而不是把类型问题整体推迟到目标语言编译器。
