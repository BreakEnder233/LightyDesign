# 复合值编辑机制

## 适用范围

1. 本文定义 WorkbookEditor 中 object、list、dictionary 等复合值的单元格编辑机制。
2. FlowChartEditor 中未来出现的复杂节点属性编辑，应复用同一套解析、类型树和递归模态规则，而不是另起一套不兼容的交互。
3. 本文只定义编辑层机制，不要求本轮立即改动 Core、DesktopHost 或工作簿 txt 文件格式。

## 当前前提

1. Sheet 单元格目前仍以原始字符串持有，强类型解析后置到 Core 的值解析阶段。
2. Core 当前稳定暴露的列类型形状主要覆盖 scalar、reference、List、Dictionary；object 尚未作为现有列类型描述器的稳定分支落地。
3. 因此复合值编辑必须把“单元格原文”和“编辑期结构化草稿”分开处理：编辑期允许识别 JSON 和结构化节点树，持久化时仍写回字符串。
4. 当前 WorkbookEditor 已有类型构造器的递归槽位编辑心智；复合值编辑器应沿用同样的“类型树 + 子模态”交互方向，而不是把全部嵌套字段一次性摊平成大表单。

## 当前落地状态（2026-04-25）

1. WorkbookEditor 已落地单元格右键“按类型编辑”入口，但当前只在精确单选单元格时显示，不对多选或整行整列选择暴露该入口。
2. number 列已同时支持行内专用数值输入和弹窗数值编辑，不再复用普通文本输入框。
3. reference、list、dictionary 已接入值编辑弹窗；list 和 dictionary 支持结构化视角与 JSON 视角切换，并支持递归打开子模态编辑当前子树。
4. reference 作为根值编辑时，编辑器会按目标表加载候选行，并按“优先 `Id`、其次 `Id1..IdN`、最后第一个可导出列”的顺序推导可选主键文本。
5. object 当前仍缺少稳定字段 schema 来源，因此现阶段只提供 JSON fallback，不提供 schema 驱动的字段级结构化编辑。
6. 结构化弹窗最终仍写回单元格字符串：number 写回数值文本，reference 写回 `[[...]]`，list / dictionary / object 写回单行 JSON。

## 设计目标

1. 当列类型是 object 或容器复合类型时，用户直接在单元格内输入 JSON，编辑器也应能正确识别、校验并保存。
2. 当列类型不适合继续依赖单行纯文本表达时，右键单元格应能打开额外编辑器，以更高效地编辑复杂内容。
3. 复合值编辑必须支持递归嵌套；每次进入子元素编辑时，都可以继续打开新的模态框处理当前子树。
4. 对 object、list、dictionary 这类复合值，行内编写格式统一收敛为 JSON，不再兼容旧的非 JSON 容器写法。
5. WorkbookEditor 与 FlowChartEditor 应共享同一套复合值编辑抽象，避免未来出现两份不同的 JSON 规则和两套不同的嵌套交互。
6. reference 视为叶子类型，不属于必须 JSON 化的复合根节点；它在 JSON 子树内部仍以字符串叶子承载。

## 非目标

1. 本文不要求把所有单元格编辑都切换为弹窗；行内编辑仍然保留，弹窗是补充能力。
2. 本文不要求兼容 List、Dictionary 的旧非 JSON 文本写法。
3. 本文不要求在本轮把 object 类型完整下沉为 Core 的正式列类型解析分支；在 Core 尚未补齐前，object 仍可先作为编辑层复合类型存在。
4. 本文不允许把完整流程图画布、超大表单或业务专用页面直接嵌入单元格内。

## 复合值模型

### 统一编辑节点

1. 编辑器应把所有复杂值统一映射为一棵 `CompositeValueSchema` 树，再基于该树创建当前单元格的 `CompositeValueDraft`。
2. `CompositeValueSchema` 的首版节点种类建议固定为五类：
3. `scalar`：基础标量叶子节点，例如 `string`、`int`、`bool`。
4. `reference`：引用叶子节点，底层仍沿用现有引用文本语义，不展开为 object、list、dictionary 等复合子树。
5. `list`：有序数组节点，包含 `elementSchema`。
6. `dictionary`：键值对节点，包含 `keySchema` 与 `valueSchema`。
7. `object`：具名字段节点，包含字段列表、字段顺序、必填约束、说明文本，以及是否允许附加字段等元数据。

### object 的定位

1. 在本机制中，object 是编辑期正式支持的复合节点，即使当前 Core 还没有把它作为稳定列类型描述器的一部分暴露出来。
2. 当 Core 尚未补齐 object 解析时，WorkbookEditor 仍可通过“表头类型 + 表头附加 schema”解析出 object 结构，并把最终值保存为 JSON 字符串。
3. 这意味着 object 的第一阶段不依赖 txt 文件格式升级；它先以“可校验的 JSON 文本 + 结构化编辑器”方式接入。

### 编辑器画像

1. 每个列类型解析结果除值 schema 外，还应带一个编辑器画像，用来决定右键菜单是否出现额外编辑器，以及打开哪一种编辑器。
2. 编辑器画像至少应回答三个问题：
3. 当前类型是否支持单元格内 JSON 直写导入。
4. 当前类型是否需要在右键菜单中暴露额外编辑器入口。
5. 额外编辑器的默认实现是“复合值类型树编辑器”，还是某个将来可扩展的专用编辑器。
6. 默认映射建议如下：
7. 普通 `scalar` 文本：默认仅保留行内编辑，不强制出现额外弹窗。
8. `number`：保留行内数值输入，同时暴露 `编辑数值...` 弹窗入口。
9. `reference`：保留行内编辑，同时暴露 `编辑引用...` 弹窗入口。
10. `list`、`dictionary`、`object`：默认同时支持行内文本输入和右键打开“复合值编辑器”。
11. 后续若某些类型虽然不是复合类型，但也不适合单行文本表达，可以通过编辑器画像声明专用弹窗，而不影响本文的复合值规则。

## 单元格内 JSON 编写规则

### 识别入口

1. 只有当列类型被解析为 list、dictionary、object 等复合值根节点时，编辑器才进入本文定义的 JSON / 结构化编辑链路。
2. 行内编辑提交、失焦校验、右键打开额外编辑器，以及粘贴文本到单元格时，都应复用同一套识别规则。
3. 编辑器不能只在弹窗里支持 JSON，而让单元格直写 JSON 继续报错；两条入口必须共享同一份解析与校验逻辑。
4. reference 属于叶子类型，不进入本节定义的复合值 JSON-only 链路；它作为根列类型时仍走自身的叶子编辑规则。

### 推荐识别顺序

1. 先按列类型拿到 `CompositeValueSchema`。
2. 若根节点是 scalar 或 reference，则走各自叶子规则，不进入本节。
3. 若当前原文为空白，则按空值 / 未填写处理，由字段约束决定是否报错。
4. 对 list、dictionary、object 根节点直接尝试 JSON 导入。
5. 若 JSON 导入成功，则按 schema 递归校验并生成结构化草稿。
6. 若 JSON 导入失败，则保留原始文本并返回可定位到字段路径的错误信息。
7. 对 list、dictionary、object 根节点，不再定义第二套非 JSON 文本语法，也不再尝试旧文本解析。

### JSON 到类型树的映射

1. `scalar` 应接受与目标类型兼容的 JSON 原始值，例如 `string` 对应 JSON 字符串，`int` / `float` 对应 JSON 数字，`bool` 对应 JSON 布尔值。
2. `reference` 在 JSON 中建议继续使用字符串承载现有引用字面量，而不是引入第二套引用 JSON 协议。
3. `list` 应映射为 JSON 数组。
4. `object` 应映射为 JSON 对象，字段名与 schema 字段名一一对应。
5. `dictionary<string, TValue>` 可接受 JSON 对象，也可接受键值对数组；两者都属于 JSON 表达。
6. `dictionary<TKey, TValue>` 当 `TKey` 不是 `string` 时，规范化 JSON 形式应为条目数组，例如 `[{"key": 1, "value": "Hello"}]`，以避免 JSON 对象只能使用字符串键导致的语义歧义。
7. 对非字符串键字典，编辑器可选地接受 JSON 对象作为额外 JSON 导入形式，但仅当每个属性名都能被安全转换为目标键类型时才允许导入。
8. `null` 不应默认视为合法值；只有字段 schema 或类型约束显式允许空值时，才允许 JSON `null` 进入草稿树。

### object 未知字段规则

1. object 类型导入 JSON 时，默认不应静默丢弃未知字段。
2. 若 schema 未声明 `allowAdditionalProperties`，未知字段应直接形成校验错误。
3. 若 schema 允许附加字段，则编辑器应把它们保留在对象草稿中，并在结构化编辑器中显式展示为“附加字段”，而不是悄悄藏起来。

### 旧写法处理原则

1. 对 list、dictionary、object 根节点，不再兼容旧的非 JSON 文本写法。
2. 如果单元格原文仍是旧写法，编辑器应直接按无效值处理，并提示改用 JSON 或打开复合值编辑器修复。
3. 结构化弹窗和 JSON 视角在点击应用后，都应一律写回稳定的单行 JSON。
4. reference 作为叶子类型，不受本节限制；它在独立列中仍可继续沿用现有引用字面量，在 JSON 内部则使用字符串承载。

### 规范化输出建议

1. 结构化弹窗写回时，应输出稳定、可比较的单行 JSON，而不是把格式化缩进后的多行 JSON 直接塞进单元格。
2. `object` 字段顺序应与 schema 字段顺序一致；附加字段若存在，应固定排在已知字段之后。
3. `list` 应保持当前显示顺序。
4. `dictionary` 应保持编辑器中的条目顺序，不要在未要求时隐式排序。
5. 任何规范化输出都不应改变值语义，也不应把合法引用值改写成另一种不兼容字面量。

## 右键扩展编辑器机制

### 触发规则

1. 当单元格所属列的编辑器画像声明存在额外编辑器时，右键菜单应增加一组“按类型编辑”操作。
2. 当前已落地的默认入口类型包括 `number`、`reference`、`list`、`dictionary`、`object`。
3. 该入口当前只在精确单选单元格时显示，避免在批量选择状态下误导用户进入单值编辑流程。
4. 即使当前单元格文本暂时不合法，右键菜单入口也不应消失；用户仍应能通过弹窗修复无效内容。
5. 右键入口只是最小保证入口，后续也可以追加双击图标、快捷键或单元格内辅助按钮，但不改变本文定义的核心行为。

### 菜单文案建议

1. object：`编辑对象...`
2. list：`编辑列表...`
3. dictionary：`编辑字典...`
4. reference：`编辑引用...`
5. number：`编辑数值...`
6. 无法细分但仍属于复合值：`编辑复合值...`
7. 若未来接入专用编辑器，则菜单文案应来自该编辑器画像，而不是硬编码到表格组件里。

### 弹窗壳层

1. 复合值弹窗应展示当前列名、当前路径、类型摘要和校验状态。
2. 弹窗内部至少应提供两个视角：
3. 结构化视角：按类型树编辑当前节点。
4. JSON 视角：直接编辑当前节点子树对应的 JSON 文本。
5. 当当前值是合法 JSON 时，弹窗应能展示其结构化结果，并允许在结构化视角与 JSON 视角间切换。
6. 如果当前原文不是合法 JSON，弹窗应保留原文并在顶部展示错误，再允许用户切换到 JSON 或结构化修复。

## 类型树与递归模态规则

### 根节点编辑

1. 根弹窗只负责当前单元格对应的复合值根节点。
2. 对象节点显示字段列表；列表节点显示元素列表；字典节点显示条目列表。
3. 根弹窗中的每个子节点都只展示摘要，不把所有后代一次性全部展开。

### 子节点递归

1. 当用户点击“编辑字段”“编辑元素”“编辑条目值”等操作时，应打开新的子模态框，而不是在当前模态里继续无限向下堆内容。
2. 每个子模态只处理当前路径对应的一棵子树，并继承同样的结构化 / JSON 双视角。
3. 子模态点击“应用”后，只把当前子树回写到父模态草稿；父模态本身是否最终提交到单元格，仍由父模态决定。
4. 子模态点击“取消”时，不应污染父模态当前草稿。
5. 模态标题中应显式展示路径，例如 `编辑对象 / reward.items[2]`，避免深层编辑时迷失上下文。

### 节点级操作建议

1. object：编辑字段值、恢复默认值、添加附加字段、删除附加字段。
2. list：插入元素、删除元素、复制元素、调整顺序。
3. dictionary：新增条目、删除条目、编辑 key、编辑 value。
4. scalar：使用与当前基础类型匹配的简单控件，例如文本框、数字框、布尔切换。
5. reference：保留现有引用语义，可继续使用更适合引用值的选择器或辅助控件。

### 校验与错误定位

1. 结构化编辑与 JSON 编辑都必须在当前子树范围内立即校验。
2. 校验错误应带路径，例如 `$.reward.conditions[1].targetId`，而不是只返回“格式错误”这类无法定位的问题。
3. 父模态在展示子树摘要时，也应能反映该子树是否存在未解决错误。
4. 只有当根节点校验通过时，才能最终写回单元格。

## 与 WorkbookEditor、FlowChartEditor 的衔接

### WorkbookEditor

1. 列类型解析结果应同时提供值 schema 与编辑器画像，供单元格行内编辑、右键菜单和弹窗编辑共用。
2. 当前实现已经直接复用宿主类型描述器构造 number、Reference、List、Dictionary 的 schema 与编辑器画像。
3. reference 候选项当前通过读取目标 Sheet 行数据并结合导出列规则在前端推导，不依赖额外的专用引用查询接口。
4. 对 object 类型，WorkbookEditor 仍需要额外的字段 schema 来源；在 Core 尚未补齐前，这一层可以先来自表头附加配置或宿主接口。
5. 结构化弹窗写回后，单元格仍只存字符串，因此不会改变现有工作簿 txt 的总体存储边界。

### FlowChartEditor

1. 节点 Inspector 中未来出现的 object、list、dictionary 复杂属性，不应在右侧栏中无限展开成超长表单。
2. FlowChartEditor 应复用与 WorkbookEditor 相同的 `CompositeValueSchema`、JSON 导入规则、规范化输出规则和递归模态机制。
3. 流程图侧即使没有“单元格右键菜单”这一入口，也应至少在属性项操作按钮或右键菜单上暴露同一类弹窗。
4. 对流程图已有的递归类型系统，后续只需新增到 `object` 或自定义复杂对象的 schema 映射，不应另造第二套 JSON 约定。

## 推荐落地顺序

1. 先定义宿主到前端的复合值 schema 与编辑器画像协议，补齐 object 的编辑层 schema 来源。
2. 再把 WorkbookEditor 的单元格行内 JSON 导入和右键弹窗入口接上统一解析链路。
3. 完成结构化弹窗、JSON 视角与递归子模态后，再决定哪些类型需要专用编辑器。
4. 最后把同一套机制复用到 FlowChartEditor 的复杂属性编辑，确保两边共享一致的值语义与 JSON 编写规则。

## 当前状态结论

1. 复合值编辑不应再停留在“把一整棵对象塞进单行文本框”的层面。
2. 当前 WorkbookEditor 已经把 number、reference、list、dictionary 的按类型弹窗编辑接上；其中 list / dictionary 支持递归子模态，reference 支持目标表候选选择。
3. 对 object、list、dictionary 这类复合值，整体方向仍是“行内输入继续保留，但编写格式统一收敛为 JSON；复杂编辑进入类型树弹窗；递归进入子树时继续开启新模态”。
4. 在 Core object 正式落地前，WorkbookEditor 先通过 JSON fallback 承接 object 编辑；等底层类型系统补齐后，再把 object 的结构化字段编辑收敛回统一协议层。
